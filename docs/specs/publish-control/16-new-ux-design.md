# 16. 新 UX デザイン — 「確認」列の廃止と単一ボタン化

Status: Proposal (2026-04-19)
Scope: `/dashboard/articles` 一覧の「確認」列を廃止し、1記事あたり 1 つの
"公開トグル" ボタンに置き換える。ボタンは押下と同時に
ハブページの再生成と関連記事の同期を原子的に行う。

---

## 1. 意思決定サマリー

**最重要決定：チェックボックスではなく「ラベル付きセグメント・ボタン
（公開中 / 非公開）」とする。**

理由:

- トグルスイッチ（iOS 風）は「オン/オフ」の意味が視覚的に曖昧。
  特に高齢ユーザ（由起子さん）にとって
  「今どっちなのか」が一目で分からない懸念がある。
- 単一ボタン（「公開する」を押すと「非公開にする」に変わる）は、
  現在状態 と 次のアクション を 1 つの UI で表す必要があり、
  認知負荷が高い（Nielsen のヒューリスティック #6 違反）。
- セグメント（pill 形状の 2-state button）であれば、
  現在の状態と取れるアクションが両方見え、色覚的にも明確。

採用: **pill 形状の 1 ボタン** + **現在状態がラベルに明示**
（例: "公開中" 緑 / "非公開" グレー）。
クリック → 確認モーダル → 実行 → 進捗 → 完了。

---

## 2. 置き換え対象と差分

| 項目 | 現状 (As-Is) | 新 UX (To-Be) |
|------|--------------|---------------|
| 「確認」列 | チェックボックス | 「公開中 / 非公開」pill ボタン |
| 「確認: 全て / 確認済み / 未確認」フィルタ | 3 ボタン | **「公開状態: 全て / 公開中 / 非公開」にリネーム**（ロジック維持） |
| タイトル横の ✅ | そのまま | 残す（ボタンと冗長だが、一覧スキャン性を確保） |
| 「サーバーに更新」ボタン | 全件まとめてデプロイ | 残す。ただし「一括公開」とは別機能として位置を整理 |
| （新規）一括操作 | なし | **行頭チェックボックスで複数選択 → 「選択した N 件を公開 / 非公開」** |

メモ:

- 「確認 (reviewed_at)」という語彙は、内部的には保持されるが
  UI 上は **「公開」** 表記に統一する。
  由起子さんの体感としては「ハブに載っている = 公開」だから。
- DB スキーマ (`reviewed_at`, `reviewed_by`) は変更しない。
  ラベルのみ変える。

補足:
- Gold (`#d4a574`) は project palette で唯一 "進行中 / 注意喚起" に
  意味的にマッチする。そのため transitioning の枠線に採用。
- 公開中の緑は palette 外だが、
  「環境（本番）に出ている」というシステム共通概念を表すために
  緑を例外的に使う。由起子さんにも馴染みがある。
  既存 `bg-emerald-500` がすでに使われているので UI 一貫性は保てる。

## 6. 確認モーダル（one-click vs modal）

**採用: モーダル必須（ただし例外あり）**

| 操作 | 確認方法 | 根拠 |
|------|----------|------|
| 非公開 → 公開 | モーダル | 本番に出る、関連記事が書き換わる |
| 公開 → 非公開 | モーダル（destructive 赤） | 本番から消える、由起子さん心理的に重い |
| 処理失敗からの再試行 | モーダル | 冪等性を自信を持って伝えるため |
| 一括操作 | モーダル（件数・件数プレビュー付） | まとめて触るほど影響範囲大 |
| **例外: 直前 15 秒以内の自分の操作の undo** | モーダルなし（トーストの "元に戻す" リンク） | 操作直後は意図が明確、モーダルは冗長 |

> 現状コード (`page.tsx` L650) は「非公開化のみ確認、公開化はノー確認」。
> 新 UX では **両方向で確認する**。理由: 公開も "本番に出る" destructive 系。
> 由起子さんのフィードバック「デグレに非常に敏感」と整合。

## 7. 進捗インジケータ

デプロイは 10–20 秒かかるため、視覚的に「動いている」ことを示す。

### 7.1 単発

- ボタン状態: `公開中 / 非公開` → `更新中…` (spinner, gold 枠)
- 行全体: `opacity: 0.85`, `pointer-events-none`
- 他行は操作可能（並列実行を妨げない）
- 20 秒以上かかる場合: ボタン下に `まもなく完了します…` のヘルパテキスト
- 60 秒タイムアウトで `失敗` 状態に遷移 + モーダル `サーバーが応答しません。再試行しますか？`

### 7.2 一括

- 上部に固定プログレスバー: `[━━━━━━▒▒▒▒] 6 / 10 件完了`
- 完了した行から順に pill の色が確定（視覚的フィードバック）
- 途中キャンセル可 (`[中断]` ボタン) — 完了済みは保持

## 8. 一括操作（Bulk）

### 8.1 選択 UI

- 表の先頭に新列 `[☐]` (checkbox) を追加
- ヘッダに「全選択」チェックボックス（現ページ内のみ）
- 1 件以上選択されると表下に **スティッキー操作バー** を表示

### 8.2 操作バー仕様

```
┌────────────────────────────────────────────────────────────────┐
│  選択中: 3 件                                                  │
│  [選択を公開] [選択を非公開] [選択解除]                        │
└────────────────────────────────────────────────────────────────┘
```

- `選択を公開` クリック → モーダル `3 件の記事を公開します。よろしいですか？`
- 既に公開中の行は選択しても **スキップ**（モーダルに「うち 1 件は既に公開中のためスキップします」と注記）
- 実行中は各行のボタンが順次 `更新中…` に変わり、プログレスバー更新

### 8.3 制約

- 1 回の一括操作は最大 50 件（サーバ負荷 & タイムアウト回避）
- 全選択した場合、50 件を超えるなら警告モーダル `50 件までしかまとめて処理できません。先頭から 50 件を処理します。`

## 9. フィルタタブの扱い

`確認: 全て / 確認済み / 未確認` → **`公開状態: 全て / 公開中 / 非公開`** にリネーム。

- 動作（`reviewFilter` state とクエリ）は完全にそのまま
- ラベルのみ置換
- 緑バッジ（現行）はそのまま維持（`bg-emerald-500 text-white`）
- 区切りの `|` も現行踏襲

内部コード（`reviewed_at`, `reviewed_by`）は変更せず、
UI 文言層のみ変更することで影響範囲を最小化する。

## 10. アクセシビリティ (WCAG AA)

パレット色のコントラスト比（WebAIM 計算値）:

| 組み合わせ | 比率 | 判定 |
|------------|------|------|
| `#53352b` on `#faf3ed` (非公開テキスト) | 10.8:1 | AAA |
| `#047857` on `#ecfdf5` (公開中テキスト) | 5.5:1 | AA pass (normal) |
| `#92400e` on `#fffbeb` (処理中テキスト) | 6.8:1 | AA pass |
| `#b91c1c` on `#fef2f2` (失敗テキスト) | 6.3:1 | AA pass |
| ボタン枠 `#8b6f5e` vs 背景 `#faf3ed` | 3.2:1 | UI component AA (3:1 以上) |

対応すべき追加事項:

1. **色だけに依存しない**: アイコン (`●` `○` `⟳` `⚠`) とラベルを併記。
2. **フォーカスリング**: `focus:ring-2 focus:ring-brand-500/40`
   （現行の他ボタンと同じトークン）。
3. **aria-label**:
   - ボタン: `aria-label="記事「{title}」の公開状態: {現在の状態}。クリックで切替"`
   - 処理中: `aria-busy="true"` + `aria-live="polite"` で状態変化を読み上げ
4. **キーボード**:
   - `Space` / `Enter` で発火
   - モーダル: `Esc` でキャンセル、`Enter` で主ボタン
   - 行クリックとのイベント競合回避のため、ボタンでは
     `e.stopPropagation()` を明示
5. **タップ領域**: モバイルで最小 44×44px（現状チェックボックス 16×16 は違反）。
6. **トースト**: `role="status"` で通知される
7. **視覚障害者向け状態通知**:
   処理完了時に `aria-live="polite"` で
   `「ペットロスとの付き合い方を公開しました」` をスクリーンリーダに通知。

## 11. モバイル挙動

- デスクトップ表の「公開」列は、モバイルカード下部右側の **pill ボタン** として配置（3.2 参照）
- タップ領域: 最小 44px 高
- モーダルはフルスクリーン（iOS safe-area 考慮、`env(safe-area-inset-bottom)` でボタン領域を底上げ）
- 一括操作: モバイルでは長押しで選択モード突入（iOS メール風）
  - 長押し → 行先頭にチェックボックス出現
  - 下部に同じスティッキーバー（モーダルではなく bottom sheet）

## 12. 空・エッジケース

| ケース | 挙動 |
|--------|------|
| `status != 'published'` | pill ボタン非表示。代わりに `公開前` グレーバッジ。ツールチップ `記事を"公開済み"ステータスにしてから公開できます` |
| ネットワーク切断 | 楽観的更新は行わず、最初から `失敗` 状態表示 |
| 連打（debounce） | ボタン押下 → 即 disabled、300ms 以内の再クリックは無視 |
| タブ離脱中に他ユーザが状態変更 | 表示復帰時（`visibilitychange`）に `fetchArticles()` で最新化 |
| 一括処理中のページ遷移 | 離脱警告 `処理中です。ページを離れると中断されます` |
| `/api/hub/deploy` のみ失敗 (記事 DB は成功) | pill は `失敗` 状態、トーストに `ハブの再生成に失敗しました。「サーバーに更新」で手動再生成できます` |
