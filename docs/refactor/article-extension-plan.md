# 短文記事の事後拡張プラン

> ステータス: 設計のみ（実装着手前）
> 対象: Harmony Column Generator / 32 記事のうち target_word_count 未達のもの
> 最終更新: 2026-05-02

## 1. 問題

- 現状 32 記事を生成済み。Stage2 投入直後の文字数分布は要再計測（**W1/W7 計測結果はここに転記**）。
  - W1: 平均 / 中央値 / 最小 / 最大 = `TBD`
  - W7: target_word_count 未達件数 = `TBD` / 32
- 多くの記事が `articles.target_word_count`（既定 2,000 字）に満たない短文で確定しており、SEO・読了体験ともに弱い。
- 再生成全件は API コスト・レビュー履歴の喪失リスクが大きいため、**部分拡張**で済ませたい。

## 2. 既存 API: `/api/articles/[id]/regenerate-segment`

| scope        | 入力範囲                | 用途                                     | 履歴 |
|:-------------|:------------------------|:-----------------------------------------|:-----|
| `sentence`   | 1 文                    | 軽微な言い回し直し                        | `article_revisions` に旧本文 INSERT |
| `chapter`    | 章（h2 ブロック単位）    | 章単位の深掘り・再構成                    | 同上 |
| `full`       | 本文全体                | 構成を含む全文再生成                      | 同上 |

- 共通動作: 旧本文を `article_revisions` に履歴 INSERT → 新本文で `articles.body_html` を上書き（HTML History Rule 準拠）。
- トークン消費は scope に比例。`full` はハルシネ・トーン崩れ・CTA 配置ズレのリスクが最大。

## 3. 拡張オプション

### 案 A: 全文再生成 (`scope='full'`)
- target_word_count を引き上げて再生成。構成自由度は最大。
- 既存トーン・比喩・CTA 配置を**全て破棄**するため、由起子 FB 14 項目との再整合が必要。

### 案 B: 既存本文 + 「より深い洞察を追加」prompt（新規 API or scope 追加）
- 既存 `body_html` を保ったまま、章間 or 末尾に深掘り段落を **追記**。
- Gemini に「現状本文を読み、不足する具体例・比喩・読者への問いかけを 800〜1,200 字で追加。語尾優しく、断定回避」と指示。
- 既存記述を壊さず文字数だけ伸ばせる一方、構成バランスが崩れる場合は再編集が必要。

### 案 C: 章ごとに展開 (`scope='chapter'` を順次適用)
- 各 h2 章を 1 件ずつ拡張。レビューしながら進められるため安全。
- 章数 × API コール数となりオーバーヘッドが大きい。バッチ化が前提。

## 4. 推奨選択基準

| 状況                                                   | 推奨 |
|:-------------------------------------------------------|:-----|
| 構成自体が破綻している / target を 1.5 倍以上引き上げる | A    |
| 構成は良いが内容が薄い・具体例不足                      | **B**（第一候補） |
| 章ごとの濃淡が大きく、特定章だけ薄い                    | C    |

> 既定方針: **案 B**。既存本文を壊さず追記のみ。A/C はユーザーが明示指示した記事のみ適用。

## 5. リスク

- **memory rule 抵触**: `feedback_preserve_article_content` により「明示指示なく既存記事の本文/タイトル/コンテキストを変更禁止」。
  - → 拡張処理の起動には**ユーザーの明示指示（記事 ID 単位 or 全件 GO）必須**。バッチ自動実行 / cron 起動は禁止。
- **履歴肥大**: `article_revisions` に旧本文を必ず INSERT（HTML History Rule）。保持は最新 4 世代のみ（既存ルール踏襲）。
- **CTA 配置のズレ**: 追記時に CTA 3 箇所配置が崩れないか検証必須。
- **トーン崩壊**: 由起子 FB 14 項目（""禁止・抽象NG・語尾優しく 等）の prompt 注入を継承。
- **関連記事への影響 (P5-59)**: 拡張対象が `generation_mode='zero'` の場合、本処理は `generation_mode` を変更しないため関連記事プール (同一 mode 内のみ) は維持される。万一拡張で生成方式を切り替える場合は `auto-related.ts` の同一モードフィルタにより自動的に関連プールから外れる点に注意。

## 6. 検証手順

1. 候補 1 記事を選定（最も短い or 最も target 未達なもの）。
2. ユーザーに記事 ID と適用案（A/B/C）を提示し**明示承認**を取得。
3. dry-run（DB 書き込みなし）で生成結果を取得 → 文字数・CTA 数・""有無・抽象表現を機械チェック。
4. 由起子トーン手動確認（必要なら本人レビュー）。
5. 承認後に本適用 → `article_revisions` 履歴 + `body_html` 更新。
6. 1 件成功後、対象記事を 3〜5 件に拡張し再評価。問題なければ全件展開可否をユーザーに諮る。
